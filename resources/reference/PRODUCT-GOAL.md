# 产品目标文档（Product Goal / Spec）

> 状态：v4（Phase 1 收敛为 MVP，见 §14；品牌名已定）
> 品牌：**WhichToUse** ｜ whichtouse.com ｜ X: @whichtouse ｜ 仓库：git@github.com:wang-chonghuan/whichtouse.git
> Tagline：*WhichToUse — the honest ranking of AI agents, by what you're doing.*
> 语言市场：英文/全球为主，中文为分发副本
> 配套：MARKET-RESEARCH.md（cap12 调研报告，证据与数据源明细）

## 0. 一句话定位

**一个按"业务用途"组织、把 app/SaaS、skill、开源项目三种形态各自排名并给出综合推荐的紧凑 agent 产品排名站；底层用爬取铺出广度基础，上层用真实测评文章与竞品比较逐条加深；靠双语自媒体做分发，免费 skill 拉信任、收费产品做 affiliate；并刻意做成 AI 可引用的来源。**

---

## 1. 要解决的问题（精确版）

AI agent 产品爆炸，想选型的人面对两种糟糕现状：

1. **自媒体是时间线式的**：KOL 今天推一个、明天推一个，不分类，信息爆炸，看完记不住、无法横向比较。
2. **目录站已经很多，但都按"产品类型"切且不可信**：
   - 商业 agent 目录（aiagentsdirectory、theaiagentindex、aiagentslist）：**只收商业化产品**，靠厂商提交/SEO/付费收录/众评，无真测，"最推荐"不可信，UI 臃肿。
   - skill/MCP 目录（awesome-claude-skills、VoltAgent、Smithery/Glama/PulseMCP、skills.sh、SkillsMP）：**存在且不少**，但按"它是什么"组织、按 star 排、不测，且与商业产品**互相隔离**。

**真正的空白 = 没有人做"按用途横切、跨形态、且真测"的排名。**
一个做营销的人想知道"我该用什么"，得分别翻四个孤岛（商业目录 / MCP 目录 / skills list / GPT store），每个按类型组织，没有一个按"我要干什么"组织，也没有一个把免费 skill 和收费 SaaS 平等放在一起真测对比。

**这个空白有结构性原因、因而难被现有玩家填补**：每个孤岛的商业模式都围绕一种产品类型（商业目录靠厂商广告、MCP 目录靠 MCP 生态），"把免费 skill 和收费 SaaS 平等对比"没人有动机做——因为免费 skill 不给他们钱。**这是别人不愿进的窗，不是没发现的窗。**

## 2. 目标用户

- **主**：想给某个具体业务场景选 AI 能力的人——开发者、独立创业者、运营/增长、技术决策者。
- **次**：AI 老用户、想追踪新品但受不了信息爆炸的人。
- **前瞻**：**别人的 agent**——用户把"帮我找最好的 X agent"委托给自己的 AI 时，我们希望成为被引用的来源。

## 3. 产品形态与信息架构（核心）

### 3.1 信息架构：用途 × 三形态 × 排名

**主呈现是"业务用途分类"**，例如：
UI 设计、找客户/获客、生成视频、做短剧、画架构图、写代码、客服陪聊、财务记账、数据分析、内容写作、SEO、研究/搜索、做 PPT、图片生成、语音音频、邮件外联、简历求职、法律合同、翻译……
（**分类清单靠综合多个竞品的分类法 + 搜索趋势得出**——抓 aiagentsdirectory / theaiagentindex / Futurepedia / TAAFT / Product Hunt / G2 的分类，归并成一套最全的 canonical 分类；**每个分类打"赚钱等级"标签**：🟢高（写作/视频/语音/营销/销售/找客户，recurring affiliate 成熟）、🟡中、🔴低（编码/开源/skill，拉信任不赚钱）；**全面收录但按赚钱等级排优先级**，最终留哪些分类放进工单由人 grill 定夺。见 §4、§14。）

每个分类下，分三条形态轨道，各自有排名，外加一个综合排名：

```
分类（如：找客户）
├── app / SaaS 排名        （Top N，条数灵活）
├── skill 排名             （Top N，条数灵活）
├── 开源项目 排名           （Top N，条数灵活）
└── 综合排名 Best 3（跨形态）  ← 见 §6 方法论
```

此外有一个全站级 **🔥 本月上升榜**（每轨 app/skill/git 各 ≤5 个近一个月升势最快的），作新品发现入口，见 §14.6。

### 3.2 交互原则

- **无 landing 页**：点进去直接是分类。
- **两次点击见答案**：首屏是分类 → 点分类 → 该用途下三轨排名 + 综合推荐。
- **紧凑、直给、言之有物**：浅色主题、小字体、密集信息（HN/维基风）；**不要**深色大图标、松散大 UI、成堆标签/卡片/折叠。
- **AI 友好**：结构化数据 / llms.txt / 简单 API，方便 agent 读取与引用。

### 3.3 两层内容模型（关键，解掉"广 vs 深"的矛盾）

| 层 | 是什么 | 怎么来 | 作用 |
|---|---|---|---|
| **基础层** | 每分类约 10 app + 10 skill + 10 开源（≈30 条，**条数灵活**：冷门分类或稀缺形态可少，热门可多） | **爬取** + 客观信号排序 | 冷启动即不薄、机器可读 |
| **深度层** | 单品评测文 + 竞品比较文 | **真测**（沙盒/浏览器/免费版） | 信任、差异化、自媒体内容 |

基础层给广度，深度层给可信度；**每一条深度评测都会替换/升级基础层里对应条目的徽章与排名依据**。

### 3.4 每条产品的数据字段

用途分类 · 形态（app/skill/开源）· 名称/链接 · 定价 · 核心能力 · GitHub star/热度（若适用）· **测评深度徽章**（🧪沙盒实测 / 🆓免费版实测 / 📋仅画像）· 我的评分与结论 · affiliate 链接（若有）· 关联的评测文/比较文 · 选题-发布记录。

## 4. 内容与增长循环

### 4.1 冷启动（一次性 bootstrap）

1. **爬分类**：爬出用户喜闻乐见的业务用途分类清单。
2. **爬基础层**：每个分类爬约 10 app + 10 skill + 10 开源（**条数灵活**，凑不齐不硬凑、富余可多收），入库、按客观信号给出**临时排名**（标 provisional）。
3. 站点上线即有广度骨架。

### 4.2 持续增长（日常循环，也是自媒体产能）

**一次选题 = 测一个产品 = 出评测文 + 比较文 = 升级该分类排名 = 一篇自媒体 = 一个引流入口。**

- 触发：出现**新的知名产品**，或**重评一个老产品**。
- 产物：一篇评测文（这东西怎样）+ 一篇竞品比较文（在同类里排第几、综合该不该选它）。
- 分发：介绍 → 与同类老产品比较 → 引流到站点试用。有多个新品就合并发一次。
- 节奏：约每天 1–2 篇，以"能真测"为上限，宁可少而真。

**内容生产 = 排名加深**：站点不是一次堆满就不动，而是被评测文一条条"从画像升级为实测"。

## 5. 测评策略（深度层，分层 + 低成本）

**不是什么都测，用 star / 成熟度做前置闸门（star-gating）：**

| 产品类型 | 测的门槛 | 测法 | 徽章 |
|---|---|---|---|
| 开源 skill / MCP / plugin | GitHub star 达阈值（借 skills.sh / SkillsMP 热度） | 沙盒 + 浏览器实跑 | 🧪 沙盒实测 |
| 有 free plan 的 webapp | 已发布且有免费额度 | 注册免费版实跑 | 🆓 免费版实测 |
| 纯付费 / 企业 SaaS | —— | 只按文档/定价/demo 画像 | 📋 仅画像 |

**诚实交代"这条我没动手测"本身就是信任杀招**——那些假装全评过的 slop 给不了。

## 6. 排名方法论（这个结构逼出来的核心设计）

**轨道内排名**（app / skill / 开源各一份）：答"在这个形态里，做这件事选哪个"。用可得客观信号（star/热度/定价/能力）+ 真测结论；未测的标 provisional。

**综合排名 Best 3**（跨形态）：答"不管形态，现在把这件事做好，最该用什么"，给出**排好序的前三名**。
- 排序轴是"**把活干好的实际效果 / 性价比**"，**形态和价格是属性、不是排序轴**（否则免费 skill 和 500 刀 SaaS 无法比）。
- Best 3 里尽量兼顾不同形态/价位，让读者一眼看到"开箱即用/免费开源/企业级"各自的头部，但**仍是一个有序的前三**（有冠军，利于自媒体传播）。
- **Phase 1 就出一版**：跨轨归一化的**算法合成 Best 3**，明确标 📋 算法初选；Phase 2 评测逐条覆盖升级。

**各轨道基础排名信号**：
- 开源 / skill：GitHub star + 使用热度（skills.sh / SkillsMP）。
- app / SaaS（无 star）：**流量 + 融资 + 评论数** 三个信号。
- 三轨的真测结论一旦产出，覆盖并升级对应条目的排名依据。

**贯穿全程的诚实原则**：排名必须**透明标注**哪些是信号排、哪些是真测。基础层多为 📋，逐步被 🧪/🆓 替换。
**已知弱点**：app/SaaS 的三信号也偏"外部画像"，早期排名标 provisional，靠后续实测收敛。

## 7. 核心差异化 / 护城河

1. **跨形态、用途优先的诚实排名 + 可辩护的综合推荐**——无人占、且现有玩家无动机占（结构性空白）。
2. **UI 优势是防御性的**：竞品臃肿是被商业模式逼的（SEO 长页、广告位、厂商提交要地方放）。我们的模式（自媒体分发 + affiliate + 真测）移除了这三个压力，因此**能**做紧凑诚实 UI，**竞品照抄就等于放弃收入**。
3. **诚实 + 真测的信任**：竞品全是提交/SEO/众评/付费收录，我们真跑、给证据、明确标测评深度。
4. **双语分发**：英文目录有结构没分发；中文 KOL 有分发没结构。我们双语 + 技术 + 人在爱尔兰，能同时干两半，两拨竞争者都复制不了另一半。
5. **成为"AI 会引用的排名"**：干净、机器可读，slop 目录 agent 读不动。既是护城河也是新分发渠道。

## 8. 商业模式

- **主变现**：**affiliate（美元）+ 自有 newsletter 受众**。平台发文是引流，不是主收入。
- **稿费/平台创作分成**：薄，不作主收入预期。
- **红线：不碰厂商赞助 / 付费收录**——一碰唯一的差异化（可信独立测评）就没了。
- **免费/收费分工**：同一用途排名里，**免费 skill/开源拉信任和流量**，**收费产品做变现（affiliate）**；正因诚实摆出免费选项，对收费产品的推荐才可信，转化反而更高。**免费内容补贴信任，收费槽位收割信任。**
- **市场与收款**：英文/全球为主战场（工具生态英文、affiliate 美元、爱尔兰欧盟账户直接收款）。中文作影响力副本（分发势能，非主要现金）。

## 10. 技术架构方向

复用 stemrobin 骨架，三块从零。

| 层 | 处理 |
|---|---|
| Web 骨架（TanStack Start SSR）/ PostgreSQL / n-easyapp 部署 / prodfarm 治理 | ✅ 照搬 |
| 内容生成 skill（源数据 → 生成 → 入库 → 渲染） | ✅ 模式照搬，源数据换成"agent 产品评测库" |
| **爬取管线**（爬分类 + 每分类 top-N 基础层，多数据源：GitHub/skills.sh/Smithery/MCP Registry/aiagentsdirectory/Product Hunt/**last30days-skill**/Radar/Tranco） | 🔴 从零，冷启动骨架 |
| **证据化调查/真测**（沙盒+浏览器实跑、采集证据、出评测文与比较文） | 🔴 从零，决定可信度、也是产能瓶颈 |
| **多平台原生发布**（每平台原生变体 + 排期 + 发布 + 回收数据） | 🔴 从零，最难、最该先验死活 |
| 自有站展示 | 复用 stemrobin 渲染 |

**数据模型核心**：分类（category）× 形态轨道（app/skill/开源）× 产品条目（§3.4 字段）× 排名（轨道内 + 综合）× 文章（评测/比较）× 选题-发布记录。

## 11. 关键风险（须直面）

1. **真测的可行性与产能**（**生死线**）：唯一护城河是可信动手测评；若只爬不测，就沦为第 N 个聚合器。很多产品要注册/付费/接真实数据，未必能自动化真测——**最该先验证**。
2. **基础层可信度 / 排名方法论**：爬来但没测的 30 条若排得不透明，就变 slop。靠徽章 + provisional 标注 + 综合排名场景化，但方法论要经得起推敲。
3. **平台 AI 内容政策 + 原创/重复检测**：跨平台群发同一篇会被判非原创限流、AI 内容可能被降权/封号。解法：每平台原生变体 + 人过一道 + 错峰，不做全自动群发。
4. **出站发布的技术脆弱性**：多数平台无公开发帖 API，只能浏览器自动化（脆弱、踩 ToS）；X API 有价格/政策风险。按平台评估、先做可行的。
5. **爬取覆盖与维护成本**：分类和基础层要持续新鲜；采集是长期维护件。
6. **数据合规（公开站）**：多个源 robots 封 bot（PulseMCP/aiagentslist/Glama）、aiagentsdirectory Terms 禁未授权采集、Product Hunt 禁商用；**人在爱尔兰=欧盟，有数据库特殊权利，整库镜像别人的库有侵权风险**。解法见 §14 数据获取原则：**多源汇编 + 只取每分类 top-N + 自有打分，不镜像任何单一竞品的库、不拖库**。
7. **"没人触达我"的解读偏差**：部分是关注列表选择偏差；英文推特工具号其实很卷。差异化仍靠"结构化+真测+站"。

## 12. 非目标 / 红线

- ❌ 不做付费收录、厂商赞助、软文。
- ❌ 不做"大而全但不测"的死目录（基础层可爬，但深度靠真测、且诚实标注）。
- ❌ 不要 slop UI：不深色、不大图标、不松散、不堆标签卡片折叠。
- ❌ 不发布未经我过目的内容（发布是对外动作，须人把关）。
- ❌ 不为产能牺牲"真测"的可信度。

## 14. 分阶段路线图（v4：Phase 1 收敛为 MVP）

### 14.1 路线图

- **Phase 1｜MVP：只有网站**（本次收敛的最小可行）——公开上线一个紧凑网站，能按业务用途看到三形态各自的头部；**少量分类、单源、原生信号排序、一次性 ingest**。目标：验证概念/UI + 拿到公开 URL + 作后续骨架。详见 §14.2。
- **Phase 2｜完整数据管线**（原 Phase 1 整体后置至此）：多源汇编 + 名次归一合成分 + 全量分类 + 分类映射表 + 本月上升榜 + cron 刷新 + 算法 Best 3 v1。详见 §14.3。
- **Phase 3｜评测 + 文章 + 分发**（原 Phase 2 后置至此）：真测随文章做（每篇选题 = 一次真测 = 升级排名 = 一篇自媒体）；affiliate、多平台发布。

> **贯穿原则：尽量简单，简单有效优先，每阶段只做该阶段最小集，不提前复杂化。**

### 14.2 Phase 1（MVP：只有网站）

**目标**：最小可行——公开可访问、紧凑、按业务用途组织、能看到三形态头部的网站。**不追求全、不追求准、不追求自动化**，只求"立起来、看得出概念、能拿去给人看"。

**范围内（做）**
- 复用 stemrobin 骨架（TanStack Start + PG + n-easyapp），**公开部署**，拿到 URL。
- **少量分类（约 5 个，手挑，偏 🟢 赚钱类）**——不爬全量 taxonomy。
- 每分类三轨（app / skill / 开源），每轨 **top ~10，来自单一最省的源**：
  - 开源 = GitHub Search API（关键词 + 按 star）。
  - skill = Smithery API（按 useCount）或 skills.sh 二选一。
  - app/SaaS = Product Hunt API 或 aiagentsdirectory `/api` 二选一。
- **按该源原生信号排序**（star / 安装量 / votes），**不做合成归一**。
- 全部标 📋 provisional。
- **紧凑、浅色、无 landing、分类优先**的 UI；结构化数据若便宜则顺带出。
- **一次性 ingest**：手动跑脚本入库即可，**不建 cron、不建自动刷新**。

**范围外（推到 Phase 2+）**
- 多源汇编、合成分/名次归一、跨源加权。
- 综合 Best 3（MVP 不做，或最多裸取各轨第一名并列）。
- 全量分类、分类映射表（MVP 少量分类手写查询词即可）。
- 本月上升榜（需 cron 攒历史）。
- last30days / Radar / Tranco / 融资等富化源。

### 14.3 Phase 2（完整数据管线，原 Phase 1 内容）

在 MVP 站立起来后，把数据从"单源原生排序"升级为"多源汇编 + 自有打分"，并补全分类与上升榜。以下为已敲定的设计（保留备用）：

**决策**：① 分类尽量全面，用多竞品综合出 canonical 分类，按"赚钱等级"🟢/🟡/🔴 分优先级，最终留哪些由人 grill；② 综合 Best 3 出算法 v1（标 provisional）；③ 出全站级"本月上升榜"（每轨 ≤5）。

**数据获取原则（合规 + 轻量）**
- **不拖库、不镜像任何单一竞品整库**；每分类 × 每源只取 top-N（≈10），跨源汇总。
- **多源汇编 + 自有打分**：多平台 + 一手源采集"事实"，灌进自己的库、自己重新组织——不逐字转载。既避开欧盟数据库特殊权利/各源 ToS，也正是产品价值。
- **优先官方 API**（GitHub / MCP Registry / Glama / aiagentsdirectory `/api` / Product Hunt）；封 bot 的走官方 API 或不用；没 API 的（skills.sh、融资新闻）才 agent 访问取前 N。
- **去重以 repo URL / 域名为主键**；跨源都上榜 = 强信号。
- **分类映射表**：canonical 分类 → 各源查询词/topic，LLM 起草人校，不现场分类。

**合成分（"自己分析"）**：各源原始数值不可直接相加 → 换成**分类内名次百分位**再合成；跨源都上榜加权；合成分 → 轨道排名 → 跨轨归一 → Best 3 v1。每轨的源：开源 = GitHub Search API + awesome-list；skill = Smithery(useCount) + skills.sh + MCP Registry；app/SaaS = Product Hunt + aiagentsdirectory `/api` + Radar/Tranco(流量) + 新闻(融资)。

**重要补充源：`last30days-skill`（MIT）** — `github.com/mvanhorn/last30days-skill`，跨平台（Reddit/HN/X/YouTube/GitHub…）按真实互动量聚合的 Claude skill，`--emit=json` 机读、有发现模式。用途：给 app/SaaS 补跨平台"热度"信号（它们没 star）；发现模式喂选题与上升榜。X/YouTube 需自带密钥，Reddit/HN/GitHub 零配置；含噪，作一个信号。

**本月上升榜**：全站级、每轨 ≤5。git/skill 用 GitHub 近 ~4 周升星增量（靠每周 cron 自采 star 快照算差值，需攒几周历史）；app/SaaS 用本月 PH 新发布高 votes + last30days 发现模式。
